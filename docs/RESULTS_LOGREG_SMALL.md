# logreg_small Result Summary

---

# English

This document summarizes the current `logreg_small` experiment.

The result is generated by:

```bash
./scripts/run_logreg_small.sh
```

## 1. Experiment

The `logreg_small` benchmark uses the following polynomial inference graph:

```text
z = 0.8*x1 - 0.5*x2 + 1.2*x3 - 0.3
y = 0.5 + 0.197*z - 0.004*z^3
T = 0.5
```

The benchmark focuses on samples near the decision threshold.

## 2. Compared Methods

The current experiment compares:

- uniform precision schedules
- accuracy-only schedules
- FlipGuard decision-margin-aware schedules

Representative methods:

```text
uniform_bits_12
uniform_bits_16
accuracy_only_tol002_m12
accuracy_only_tol0005_m16
flipguard_p5_m12
flipguard_p1_m16
```

## 3. Main Result

Current paper-ready result:

```text
flipguard_p5_m12:
  stable boundary flips = 0
  certification = p5
  usage = 0.94x
  bound/max = 3.71x
  average bits = 9.09
  saving vs U12 = 24.24%

flipguard_p1_m16:
  stable boundary flips = 0
  certification = p5 + p1
  usage = 0.97x
  bound/max = 2.77x
  average bits = 11.36
  saving vs U16 = 28.98%
```

## 4. Interpretation

The current result suggests that FlipGuard can reduce average scheduled precision while satisfying decision-margin certification.

Accuracy-only scheduling can sometimes avoid stable-boundary flips empirically, but it may fail decision-margin certification. In the current result:

```text
accuracy_only_tol0005_m16:
  stable boundary flips = 0
  certification = none
  p1 usage = 10.06x
```

This means that it empirically avoids flips in the current sample set, but its estimated error exceeds the p1 decision-margin budget.

## 5. Bound Conservatism

FlipGuard uses a sufficient error bound based on node-wise sensitivity and scheduled quantization error.

Current observed conservatism:

```text
flipguard_p5_m12:
  bound/max = 3.71x

flipguard_p1_m16:
  bound/max = 2.77x
```

This indicates that the sufficient bound is conservative but still within a measurable finite range in the current toy benchmark.

## 6. Limitation

This result is based on a controlled plain simulation benchmark.

It does not yet measure:

- real CKKS ciphertext execution
- scale/rescale behavior
- level consumption
- runtime
- bootstrapping impact
- real dataset behavior

These will be added in later stages.
